The `@utexo/rgb-sdk` package is the Node.js SDK for applications built on the Utexo stack. It provides a complete set of bindings for wallet management, RGB asset operations, Lightning payments, async payments, and on-chain Bitcoin interactions. It is built on top of `@utexo/rgb-sdk-core`, the shared foundation used by all three Utexo SDK platforms.

All operations are performed locally using client-side validation. Private keys and mnemonics are never transmitted to remote servers during normal execution.

<Warning>
  This SDK is designed for **Node.js only** and is not browser-compatible. For browser environments use `@utexo/rgb-sdk-web`; for iOS and Android use `@utexo/rgb-sdk-rn`.
</Warning>

## Installation

```bash
npm install @utexo/rgb-sdk
```

## Initialisation

The primary wallet class is `UTEXOWallet`. Initialise it with a mnemonic and a configuration object, then call `await wallet.initialize()` before use.

```js
const { UTEXOWallet, generateKeys } = require('@utexo/rgb-sdk');

// 1. Generate wallet keys
const keys = await generateKeys('utexo');
console.log('Mnemonic:', keys.mnemonic); // Store securely

// 2. Initialise wallet
const wallet = new UTEXOWallet(keys.mnemonic, {
  network: 'utexo',       // 'mainnet' | 'testnet' | 'utexo'
  dataDir: './wallet',
  lspBaseUrl: '...',      // optional — enables async (offline) payments
  lspBearerToken: '...'   // optional — required when lspBaseUrl is set
});
await wallet.initialize();

// 3. Get a deposit address
const address = await wallet.getAddress();

// 4. Check BTC balance
const balance = await wallet.getBtcBalance();
```

`lspBaseUrl` and `lspBearerToken` are optional. Omitting them has no effect on synchronous payment flows — async payment capability is strictly opt-in.

## Networks

| Environment | Identifier | RGB Transport Endpoint | Bitcoin Indexer |
|---|---|---|---|
| Mainnet | `mainnet` | `rpcs://rgb-proxy-mainnet.utexo.com/json-rpc` | `ssl://electrum.iriswallet.com:50003` |
| Testnet | `testnet` | `rpcs://rgb-proxy-testnet3.utexo.com/json-rpc` | `ssl://electrum.iriswallet.com:50013` |
| Utexo (Signet) | `utexo` | `rpcs://rgb-proxy-utexo.utexo.com/json-rpc` | `https://esplora-api.utexo.com` |

Pass the identifier string in the `network` field of the init config. `utexo` is the default environment for development and testing.

## Wallet Methods

<AccordionGroup>
  <Accordion title="Key Generation & Derivation">
    - `generateKeys(network?)` — Generate new wallet keys (returns mnemonic, xpubs, master fingerprint)
    - `deriveKeysFromMnemonic(network, mnemonic)` — Derive wallet keys from an existing mnemonic
    - `deriveKeysFromSeed(network, seed)` — Derive wallet keys from a BIP39 seed
    - `getXpub()` — Get vanilla and colored xpubs
  </Accordion>
  <Accordion title="Wallet Initialisation & State">
    - `getAddress()` — Get deposit address; supports address reuse with keychain rotation (addresses remain stable while keys rotate underneath)
    - `getBtcBalance()` — Get on-chain BTC balance (async)
    - `refreshWallet()` — Sync and refresh wallet state
    - `listUnspents()` — List unspent UTXOs
    - `listAssets()` — List RGB assets held
    - `getAssetBalance(assetId)` — Get balance for a specific asset
  </Accordion>
  <Accordion title="Backup & Restore">
    - `restoreUtxoWalletFromBackup({ backupPath, password, targetDir })` — Restore wallet from a file backup
    - `restoreUtxoWalletFromVss({ mnemonic, targetDir, config?, vssServerUrl? })` — Restore wallet from VSS cloud backup
    - `createBackup({ backupPath, password })` — Create an encrypted wallet backup
  </Accordion>
  <Accordion title="Transaction History">
    - `listTransactions()` — List BTC-level transactions
    - `listTransfers(assetId?)` — List RGB transfer history for an asset
  </Accordion>
</AccordionGroup>

## RGB Asset Methods

<AccordionGroup>
  <Accordion title="Asset Issuance">
    Issue a new Non-Inflationary Asset (NIA) on RGB:

    ```js
    const asset = await wallet.issueAssetNia({
      ticker: 'USD Token',
      name: 'USDT',
      amounts: [1000, 500],
      precision: 6
    });
    ```
  </Accordion>
  <Accordion title="Receiving Assets (Blind & Witness Invoices)">
    Generate a blinded or witness UTXO invoice for receiving:

    ```js
    // Blinded invoice
    const receiveData = await receiverWallet.blindReceive({
      assetId,
      amount: 100,
      minConfirmations: 3,
      durationSeconds: 2000
    });

    // Witness invoice
    const receiveData = await receiverWallet.witnessReceive({
      assetId,
      amount: 100,
      minConfirmations: 3,
      durationSeconds: 2000
    });
    ```

    - `blindReceive({ assetId, amount, minConfirmations?, durationSeconds? })` — Generate blinded UTXO for receiving
    - `witnessReceive({ assetId, amount, minConfirmations?, durationSeconds? })` — Generate witness UTXO for receiving
  </Accordion>
  <Accordion title="Sending Assets">
    Complete an RGB asset transfer (begin → sign → end):

    ```js
    // Receiver: create invoice
    const receiveData = await receiverWallet.blindReceive({ assetId, amount: 100,
      minConfirmations: 3, durationSeconds: 2000
    });

    // Sender: complete send
    await senderWallet.send({
      invoice: receiveData.invoice,
      assetId,
      amount: 100
    });

    // Refresh both wallets
    await senderWallet.refreshWallet();
    await receiverWallet.refreshWallet();
    ```

    - `send({ invoice, assetId, amount, witnessData? })` — Complete send (begin → sign → end)
    - `signPsbt(psbt, mnemonic?)` — Sign a PSBT using wallet mnemonic (async)
  </Accordion>
</AccordionGroup>

## UTXO Management

Create UTXOs in a single atomic call (begin → sign → end):

```js
const count = await wallet.createUtxos({ num: 5, size: 1000 });
await wallet.syncWallet();
```

- `createUtxos({ num?, size?, upTo? })` — Create UTXOs (async; combines begin, sign, end)

## Lightning Methods

<AccordionGroup>
  <Accordion title="Creating Lightning Invoices">
    - `createLightningInvoice(params)` — Create a standard BOLT11 Lightning invoice for receiving
  </Accordion>
  <Accordion title="Paying Lightning Invoices">
    Synchronous payments follow a begin → sign → end flow:

    - `payLightningInvoiceBegin(params)` — Start Lightning payment (returns unsigned PSBT)
    - `payLightningInvoiceEnd(params)` — Finalise Lightning payment with signed PSBT
    - `payLightningInvoice(params, mnemonic?)` — Complete Lightning payment in a single call (begin → sign → end)
  </Accordion>
  <Accordion title="Async Payments (HODL Invoices)">
    Async payments allow funds to be sent to an offline recipient. The LSP holds the payment via a HODL invoice until the recipient comes online to collect. This requires `lspBaseUrl` and `lspBearerToken` to be set in the wallet init config.

    The inbound leg (receiving while offline) is fully operational. The outbound leg (initiating a HODL payment as sender) is in active development.

    A payment hash reuse guard is enforced to prevent the same hash from being claimed across both inbound and outbound flows.

    <Note>
      HODL invoice support is being integrated into the Node.js SDK via the UniFFI bindings layer. Refer to release notes for method availability as this ships.
    </Note>
  </Accordion>
  <Accordion title="Lightning Addresses">
    When LSP integration is configured, recipients receive a stable, human-readable Lightning Address rather than a per-transfer invoice or QR code. The address remains consistent across payments while the underlying invoice and routing state are managed by the LSP.

    Lightning Addresses are provisioned automatically when `lspBaseUrl` is set — no additional method calls are required to activate them.
  </Accordion>
  <Accordion title="Payment Status">
    - `getLightningSendRequest(lnInvoice)` — Get status of a Lightning send request
    - `getLightningReceiveRequest(lnInvoice)` — Get status of a Lightning receive request
    - `listLightningPayments()` — List all Lightning payments
  </Accordion>
</AccordionGroup>

## On-chain Methods

- `onchainReceive(params)` — Generate an invoice for depositing from mainnet to UTEXO
- `onchainSend(params, mnemonic?)` — Complete an on-chain withdrawal (begin → sign → end)
- `getOnchainSendStatus(invoice)` — Get the status of an on-chain withdrawal

## Security

By using this SDK, developers have full control over transfer orchestration, UTXO selection, invoice lifecycle, and signing policy.

The SDK supports an **external signer** architecture: key management can be handled entirely outside the node, keeping signing logic isolated from the rest of the application. This is the recommended approach for custody-grade and enterprise deployments where key isolation is a hard requirement.

<Info>
  Private keys and mnemonics should be stored securely and never logged or transmitted. Use environment variables or a secrets manager to handle mnemonic storage:

  ```js
  const keys = await generateKeys('utexo');
  const mnemonic = keys.mnemonic; // Sensitive — protect at rest

  // Store mnemonic securely for later restoration
  const storedMnemonic = process.env.WALLET_MNEMONIC;
  const restoredKeys = await deriveKeysFromMnemonic('utexo', storedMnemonic);
  ```
</Info>
