Utexo is built on a layered stack. Each layer has a distinct and non-overlapping role. Together they deliver the properties that make Bitcoin-native stablecoin settlement viable for production payment systems: deterministic costs, settlement privacy, and Bitcoin-backed security. The Bridge layer extends this stack outward, connecting external EVM-based networks to the Bitcoin RGB layer.

## Stack Overview

| Layer | Role | What Utexo Uses It For |
| --- | --- | --- |
| Bitcoin | Settlement anchor and finality | Double-spend prevention, dispute resolution, irreversible state |
| Lightning Network | Off-chain execution and value transfer | High-throughput payments, near-instant finality, predictable latency |
| RGB | Asset issuance and client-side validation | Stablecoin creation, privacy-preserving transfers, Bitcoin-secured ownership |
| Utexo | Execution and coordination layer | Routing, liquidity, fee management, SDK/API abstraction |
| Bridge | Cross-chain connectivity | Atomic, non-custodial inbound and outbound transfers between Ethereum, Tron, and the RGB layer |

In the Utexo model, ownership proofs are validated locally by users' wallets (client-side validation), while actual payments occur through bidirectional Lightning channels. There is no global state to synchronise. Each transaction is a direct peer-to-peer exchange verified by cryptographic commitments.

## Bitcoin

Bitcoin serves as the settlement and finality anchor for the entire Utexo system.

It handles correctness, finality and dispute resolution, acting as the foundational layer that grants irreversible settlement and strong security. This allows Utexo to decouple settlement guarantees from execution cost volatility.

**What this means in practice:**

- Individual payments, asset transfers, and balances are not recorded on-chain. Only compact cryptographic commitments derived from Utexo activity are anchored to Bitcoin in the form of UTXOs, indistinguishable from other ordinary metadata.
- By anchoring state commitments to Bitcoin via RGB, Utexo ensures that execution taking place off-chain cannot be arbitrarily rewritten or manipulated without detection.
- This model minimises on-chain footprint while preserving the ability to prove that off-chain execution followed the agreed rules, remaining accountable to a globally verifiable settlement layer.

<Info>
  If off-chain execution were to fail, stall, or behave maliciously, participants retain the ability to fall back to Bitcoin-backed guarantees. This approach allows Utexo to scale payment activity without compromising on the long-term trust model that Bitcoin provides.
</Info>

## Lightning Network

The Lightning Network provides off-chain execution and value transfer for the Utexo system, enabling high-throughput payments and near-instant finality. Execution costs remain stable regardless of base-layer congestion.

Lightning serves as the execution path through which BTC and RGB-based assets are exchanged between participants while remaining enforceable by Bitcoin. Value transfer occurs as direct peer-to-peer state updates: execution is finalised off-chain and only falls back to Bitcoin in dispute scenarios. Transactions remain fast and deterministic, while settlement security ultimately derives from the base layer.

**What Utexo adds:**

- Within Utexo, Lightning is abstracted through a state-channel layer that manages liquidity management, routing, and failure recovery automatically.
- Utexo provides pooled liquidity and routing access through APIs, transforming Lightning into a scalable backend execution engine without requiring users to open, balance, and maintain individual channels.
- By removing transaction execution from Bitcoin's fee auction and abstracting Lightning's channel management away from users, Utexo can offer stablecoin execution that behaves like traditional payment rails: fixed latency, predictable fees, and continuous availability.

## RGB

RGB is a smart contract protocol on Bitcoin designed for privacy-preserving asset issuance and cost optimisation. It enables stablecoins to operate natively on Bitcoin without introducing global state or on-chain execution.

RGB defines how assets such as USDT are created, transferred, and validated while remaining anchored to Bitcoin's security model. It uses a client-side validation architecture, where ownership proofs and state transitions are verified locally by participants rather than recorded on a shared public ledger.

**Privacy properties:**

- Only sender and receiver hold the transaction proofs. No public ledger of individual transfers exists, eliminating data traceability.
- For operators, stablecoin balances, transfer amounts, and counterparties are not exposed on public ledgers, reducing competitive intelligence leakage while maintaining cryptographic auditability.

**Fee properties:**

- As RGB doesn't require blockspace for asset execution or state updates, stablecoin transfers do not compete for on-chain throughput.
- This removes fee volatility tied to block congestion and allows Utexo to offer predictable and stable execution costs, independent of Bitcoin mempool conditions.

For a more detailed overview of RGB's architecture, see the [RGB official documentation](https://docs.rgb.info/).

## Utexo Execution Layer

Utexo provides a payment execution environment for USDT on Bitcoin, exposing stablecoin transfers and settlement as predictable, production-ready primitives through a REST API and SDK.

Utexo handles execution routing, liquidity access, retries, and failure recovery on behalf of users, allowing PSPs and financial platforms to support Bitcoin-native stablecoin payments without embedding protocol-specific logic into their systems.

**Performance characteristics:**

- By combining RGB's client-side validation architecture with Lightning's routing network, Utexo achieves transaction times of around 200 ms latency, comparable to Visa speeds.
- Utexo makes Bitcoin usable as a stablecoin settlement network for high-throughput, private-by-default payments that behave predictably regardless of network conditions.

**Integration surface:**

- REST API and SDK for application-level integration
- Abstracts all channel management, routing, and liquidity operations internally
- No protocol-specific knowledge required in application code

## Bridge

The Utexo Bridge extends the stack outward by enabling atomic, non-custodial transfers of USDT between external EVM-based networks (Ethereum, Tron) and the Bitcoin RGB layer. It is composed of three coordinated components: the Bridge orchestrator, stateless chain Connectors, and the Enclave Signer.

### Bridge Orchestrator

The Bridge orchestrator is the central coordinator. It tracks all in-flight transfers in a persistent database, processes `FundsIn` events reported by Connectors, and instructs the appropriate destination Connector to issue a `FundsOut` transaction. Once the destination Connector reports finality, the orchestrator marks the transfer as complete.

If a transfer becomes overdue or the source-side `FundsIn` event is not registered within a validity window, the orchestrator queues a cancellation and returns funds to the user.

### Connectors

Each supported blockchain has its own stateless Connector — a gRPC microservice that performs four tasks:

1. **Read events** — Monitors the chain's Bridge smart contract for `FundsIn`, `FundsOut`, and `FundsIn Cancelled` events.
2. **Send transactions** — Issues `FundsOut` transactions on instruction from the orchestrator.
3. **Generate signatures** — Produces the cryptographic signatures required by the destination chain's smart contract to verify the transfer intent.
4. **Report chain state** — Provides block height, confirmation counts, and token balance information to the orchestrator.

Connectors are stateless: all transfer state is held by the orchestrator. If a Connector restarts, it resumes operation without data loss.

Currently supported networks: Ethereum (EVM, secp256k1), Tron (EVM-compatible), Bitcoin RGB layer (Taproot / Schnorr).

### Enclave Signer

The Enclave Signer is the cryptographic trust anchor of the Bridge. It is a Rust microservice (`utexo-bridge-enclave-signer`) running inside an **AWS Nitro Enclave** — a hardware-isolated Trusted Execution Environment (TEE) with no persistent storage, no shell access, and no network access except a vsock proxy allowlisted to an Esplora indexer.

Private keys are generated inside the enclave using a BIP-39 mnemonic derived from OS entropy and never leave the hardware boundary. The enclave derives separate key paths for each chain type:

- **EVM keys** — Derived at `m/44'/60'/0'/0/0`. Used to sign EIP-712 typed data for EVM-bound `FundsOut` authorisations.
- **Bitcoin keys** — Derived at `m/84'` (SegWit legacy) and `m/86'` (Taproot). Used to sign PSBTs for RGB → EVM transfers using Taproot Schnorr script-path signatures.

Before signing any transaction, the enclave validates the RGB consignment against an Esplora indexer in-enclave, cross-checking the RGB asset amount against the calldata amount plus commission, verifying the consignment hash for tamper detection, and checking the deadline and chain ID.

**Security properties:**

- Keys never exist outside the TEE boundary under any circumstances.
- Every signing operation is preceded by in-enclave RGB validation — the enclave will refuse to sign if amounts or hashes do not match.
- Secrets are zeroized on drop in memory (`SecretBox`). The codebase enforces `#![deny(unsafe_code)]`.
- The enclave is built with release hardening (`opt-level="z"`, LTO, `panic="abort"`) and no persistent storage — a restarted enclave generates a fresh key unless a seed is explicitly imported via a gated build feature disabled in production.

<Info>
  The Enclave Signer is the reason the Utexo Bridge is non-custodial. No operator, server or cloud environment - including AWS itself - can access the private keys used to authorise bridge transfers. The only way to issue a valid `FundsOut` transaction is via a signing request processed inside the enclave.
</Info>

### Transfer Flow

The following describes the complete atomic transfer sequence for an Ethereum → Bitcoin (RGB) transfer:

1. The user submits a deposit transaction to the Ethereum Bridge smart contract (`FundsIn`).
2. The Ethereum Connector detects the `FundsIn` event on-chain and reports it to the Bridge orchestrator.
3. The orchestrator validates the event, looks up the destination chain, and instructs the Bitcoin Connector to prepare a `FundsOut` transaction.
4. The Bitcoin Connector requests a signature from the Enclave Signer. The enclave validates the RGB consignment and, if all checks pass, signs the PSBT.
5. The signed transaction is broadcast to the Bitcoin network.
6. Once finality is confirmed, the orchestrator marks the transfer complete and TUSDT appears in the recipient's IRIS Wallet.

The reverse direction (RGB → Ethereum) follows the same orchestration pattern but swaps the signing method: the enclave signs an EIP-712 typed data payload for the Ethereum Connector instead of a PSBT.
